### [2026-05-21] BUG-001 — Botão de notificações não disparava prompt de permissão

**Sintoma:** Ícone 🔔 visível no header mas clique não abria prompt de permissão.

**Causa raiz:** `ativarNotificacoes` chamava `pedirPermissaoNotificacao()` (função auxiliar async), que por sua vez chamava `Notification.requestPermission()`. Browsers como iOS Safari exigem que `requestPermission()` seja chamado diretamente do handler de clique do usuário — a cadeia `onClick → useCallback → função auxiliar → requestPermission()` quebra esse requisito. Nenhum erro visível no console.

**Impacto:** Usuário não conseguia ativar notificações manualmente. Botão 🔕 era `disabled`, sem instrução.

**Solução aplicada:** Chamar `Notification.requestPermission()` diretamente dentro do `useCallback`, sem passar por função auxiliar. Botão 🔕 (denied) agora mostra instrução iOS/Android ao clicar.

**Risco futuro:** Em PWAs iOS instalados, o Safari pode não suportar a Notification API — tratar `'unsupported'` sempre.

### [2026-05-21] BUG-002 — Compromissos fixos do onboarding não apareciam na aba Rotina

**Sintoma:** Usuário informava "trabalho das 6h às 15h de segunda a sexta" no onboarding, mas a aba Rotina ficava vazia.

**Causa raiz:** `concluirOnboarding` salvava os compromissos fixos apenas na `memoria.rotina.comprometida` como array de strings (texto legível para a Flora), mas não criava objetos estruturados no `plano`. `RoutineView` e `WeekView` leem `plano?.compromissos || []` — com `plano = null`, a rotina fica vazia.

**Impacto:** Funcionalidade core quebrada — o principal valor do onboarding (pré-popular a rotina) não funcionava.

**Solução aplicada:** Ao concluir o onboarding com compromissos fixos preenchidos, salvar uma flag `fluxo_compromissos_fixos_pendentes` no localStorage. Um `useEffect` detecta a flag após o app renderizar e envia automaticamente uma mensagem `[Sistema]` para a Flora criar os compromissos recorrentes no plano usando linguagem natural (mais confiável que regex).

**Risco futuro:** Se o servidor estiver offline no momento do onboarding, a mensagem falha silenciosamente. Considerar retry ou fallback local no futuro.

### [2026-05-21] BUG-003 — "Token inválido ou expirado" após onboarding

**Sintoma:** Usuário completa onboarding e tenta conversar com a Flora — recebe erro 401 "Token inválido ou expirado".

**Causa raiz:** `getAuthHeaders()` chamava `supabase.auth.getSession()` sem verificar o tempo de expiração do token. Durante o onboarding o token pode estar prestes a vencer (< 60s) ou já ter vencido. Nenhum refresh era feito antes de enviar o token — o backend rejeitava com 401. Sem retry, o erro chegava direto ao usuário.

**Impacto:** Funcionalidade core quebrada — o primeiro uso após login/onboarding falhava silenciosamente ou exibia mensagem de erro técnica.

**Solução aplicada:**
1. `getAuthHeaders()` agora verifica `session.expires_at`: se token expira em < 60s (ou se não há sessão), chama `supabase.auth.refreshSession()` antes de retornar o header.
2. `fetchComAuth()` — nova função que envolve todos os fetch autenticados: se receber 401, tenta refresh + retry automático uma única vez.
3. Todos os fetch que usavam `getAuthHeaders()` diretamente foram migrados para `fetchComAuth()`.
4. `server/middleware/auth.js` agora loga o erro exato do Supabase e retorna códigos de erro (`NO_TOKEN`, `INVALID_TOKEN`, `USER_NOT_FOUND`, `AUTH_ERROR`) para facilitar debug futuro.

**Risco futuro:** O retry é feito apenas uma vez. Se o refresh falhar (sessão completamente inválida), o usuário precisará refazer o login manualmente — comportamento correto e esperado.

### [2026-05-21] BUG-004 — Loop infinito de autenticação após login com Google

**Sintoma:** Usuário faz login com Google, começa o onboarding, e é redirecionado de volta à tela de login.

**Causa raiz:** Três problemas sobrepostos:
1. `carregarDadosUsuario` disparado duas vezes em paralelo: `getSession().then()` e `onAuthStateChange` ambos a chamavam sem deduplição. No fluxo OAuth, Supabase dispara `INITIAL_SESSION` → `SIGNED_IN` em sequência. A primeira chamada, se recebia 401 (token ainda processando), chamava `refreshSession()` → disparava `TOKEN_REFRESHED` → terceira chamada → loop.
2. `onAuthStateChange` ignorava o tipo de evento (`_event`). Quando o evento era `SIGNED_OUT` ou `INITIAL_SESSION` com `session = null` (antes do OAuth terminar), `setSessao(null)` era chamado, mandando o usuário de volta ao login no meio do fluxo OAuth.
3. `redirectTo={window.location.origin}` no Login.jsx retornava URL sem barra final, que pode ser rejeitada pelo Supabase dependendo da configuração de Redirect URLs.

**Solução aplicada:**
- `getSession().then()` agora apenas libera `carregandoAuth` — não chama mais `carregarDadosUsuario`
- `onAuthStateChange` filtra por evento: só chama `carregarDadosUsuario` em `SIGNED_IN` e `INITIAL_SESSION` (com sessão presente). Trata `SIGNED_OUT` explicitamente zerando todo o estado.
- `carregarDadosUsuario` protegido com `carregandoDadosRef` — chamadas simultâneas são descartadas (guard in-flight). Erros 401 são logados mas nunca causam logout.
- `redirectTo` corrigido para incluir barra final (`${window.location.origin}/`) com fallback SSR-safe.

**Configuração obrigatória no Supabase (painel → Authentication → URL Configuration):**
- Site URL: `https://fluxo-app-zeta.vercel.app`
- Redirect URLs: `https://fluxo-app-zeta.vercel.app/`, `https://fluxo-app-zeta.vercel.app/**`, `http://localhost:5173`, `http://localhost:5173/**`

**Risco futuro:** Os `console.log` de debug foram mantidos intencionalmente para facilitar diagnóstico em produção. Remover quando o fluxo estiver estável.

### [2026-05-22] BUG-005 — Token inválido ao chamar a Flora após login com Google

**Sintoma:** Login funciona, sessão Supabase existe no cliente, mas ao enviar mensagem para a Flora o backend rejeita com 401 "Token inválido ou expirado".

**Causa raiz:** Race condition: imediatamente após o OAuth redirect, `getSession()` pode retornar `null` (sessão ainda propagando). `getAuthHeaders()` interpretava isso como ausência de sessão e tentava `refreshSession()` — que também falhava porque o token ainda não estava disponível localmente. O resultado era uma requisição sem header `Authorization` ou com token inválido.

Problema secundário: o middleware `autenticarUsuario` estava no endpoint SSE (`/api/processar/stream`), mas o timing entre `res.setHeader()`, `res.flushHeaders()` e a validação do token tornava difícil diagnosticar onde exatamente ocorria a falha.

**Solução aplicada:**
1. Validação de token movida para inline no handler SSE — elimina ambiguidade de middleware e garante que a auth ocorre ANTES de `res.flushHeaders()`.
2. Logs de debug adicionados em 3 camadas:
   - `[DEBUG]` em `enviarMensagem` (App.jsx): mostra os primeiros 30 chars do header Authorization
   - `[SSE]` no handler SSE (server/index.js): mostra presença do header, token (20 chars) e resultado do `getUser`
   - `[AUTH]` no middleware `autenticarUsuario` (server/middleware/auth.js): token (20 chars) e resultado do `getUser`
3. Handler SSE agora retorna 401 puro (JSON) se auth falhar — antes de qualquer `res.flushHeaders()`.

**Risco futuro:** Logs de debug foram mantidos intencionalmente. Remover `[DEBUG]`, `[SSE]` e `[AUTH]` quando o fluxo estiver estável em produção.

### [2026-05-22] BUG-007 — Flora citava compromissos passados como futuros

**Sintoma:** Flora às 07h13 dizia "você tem academia às 5h30 e trabalho às 6h30" — ambos já haviam passado e estavam concluídos.

**Causa raiz:** `_formatarAgendaDia` em `server/services/flora.js` formatava todos os compromissos do dia de forma idêntica, sem nenhuma indicação de se já haviam ocorrido. A Flora não tinha como saber o que já tinha passado vs o que ainda estava por vir.

**Impacto:** Flora soava equivocada e desconexa da realidade do usuário. Citava como futuro eventos que já tinham acontecido há horas, ignorando concluídos.

**Solução aplicada:**
1. Nova função `_classificarCompromisso` em `flora.js`: compara hora do compromisso (+ duração) com hora atual em minutos. Retorna `'passado'`, `'agora'` ou `'futuro'`.
2. `_formatarAgendaDia` atualizada: aceita `hojeStr` e `horaAtualMinutos`. Para o dia de hoje, adiciona `[JÁ PASSOU] ✅ concluído` / `[JÁ PASSOU] ⏳ pendente` / `[AGORA]` na linha de cada compromisso. Dias futuros permanecem sem marcador.
3. Novo bloco `blocoContextoTemporal` no system prompt com regras obrigatórias de comportamento por status: perguntar "como foi?" para passados, reconhecer "agora", citar normalmente o futuro. Ordem obrigatória ao resumir o dia: passado → agora → futuro.

**Risco futuro:** Classificação usa hora do servidor (UTC ou timezone do Railway). Se o servidor estiver em UTC e o usuário em UTC-3, o cálculo pode errar. Monitorar quando houver usuários em fusos diferentes.

### [2026-05-22] BUG-006 — Dados não sincronizados entre dispositivos (tabelas Supabase vazias)

**Sintoma:** Usuário faz onboarding em um dispositivo, abre o app em outro dispositivo com a mesma conta Google e refaz o onboarding do zero. Tabelas `planos` e `memorias` no Supabase estavam vazias após conversas com a Flora.

**Causa raiz:** Três problemas independentes sobrepostos:

1. **Perfil nunca salvo no Supabase durante o onboarding:** `concluirOnboarding` salvava apenas no `localStorage`. Ao logar em novo dispositivo, `carregarDadosUsuario` via `semDadosNoSupabase = true` → chamava `migrarLocalStorageParaSupabase` → mas o novo dispositivo tem localStorage vazio → nada migrado → usuário refazia onboarding.

2. **Shadow variable no handler SSE destruía `userId` antes do save:** `const userId = req.userId` (linha 315, commit adc1932) redeclarava a variável como `undefined` porque o SSE handler usa validação inline (não middleware) e `req.userId` nunca é setado. O save silenciosamente recebia `userId = undefined`. Corrigido no commit `da3f8a4` — mas os saves ainda falhavam por causa do ponto 1 (perfil ausente → sem dados para carregar → loop).

3. **`carregarDadosUsuario` usava condição fraca:** `if (dados.perfil)` verificava apenas se o objeto existia, não se tinha dados úteis (`nome`). Poderia aceitar um perfil vazio `{}` como "perfil encontrado".

**Impacto:** Funcionalidade core quebrada — sincronização multi-dispositivo não funcionava para nenhum usuário.

**Solução aplicada:**
1. `concluirOnboarding` agora chama `fetchComAuth('/api/usuario/salvar-perfil')` ao final (fire-and-forget — não bloqueia a transição de tela).
2. Handler SSE: logs `[SAVE]` adicionados antes e depois de cada save; saves protegidos por `if (userId)`; `salvarPerfil` removido do loop SSE (responsabilidade do endpoint dedicado).
3. `carregarDadosUsuario`: condição `semDadosNoSupabase` usa `dados.perfil?.nome`; load do perfil usa `if (dados.perfil?.nome)`; plano usa `if (dados.plano?.compromissos || dados.plano?.tarefas)`; log `[LOAD]` adicionado.
4. `server/services/dadosUsuario.js`: log `[LOAD]` mostrando o que foi encontrado por userId e erros por tabela.

**Risco futuro:**
- Se o servidor estiver offline no momento do onboarding, o save do perfil falha silenciosamente. O usuário terá que refazer onboarding na próxima sessão (não é um problema crítico — o onboarding é rápido).
- O histórico de display não é salvo no Supabase (só o histórico da API). Conversas anteriores não aparecem em novos dispositivos — comportamento aceitável por ora.

### [2026-05-24] BUG-008 — EstadoSemana: análise da IA nunca rodava (401 silencioso)

**Sintoma:** Card "Estado da semana" sempre mostrava fallback baseado em score (ex: "Fluindo bem", "Semana moderada") — nunca uma análise personalizada da IA. Sem erro visível para o usuário.

### [2026-05-24] BUG-009 — Flora inventava compromissos baseada em blocoRotina (narrativa) em vez da agenda estruturada

**Sintoma:** Flora afirmava compromissos em dias errados ou inexistentes. Exemplo real: usuário tinha redação na quarta, Flora afirmou que estava na terça. Flora usava texto narrativo do `blocoRotina` como se fosse agenda atual.

**Causa raiz:** Quatro problemas sobrepostos em `server/services/flora.js`:

1. **Ausência de regra explícita contra inventar compromissos.** As REGRAS ABSOLUTAS do `blocoAgenda` diziam "confie na agenda" mas não proibiam explicitamente criar compromissos a partir do `blocoRotina` ou da memória narrativa.

2. **Ordem errada dos blocos no prompt.** `blocoMemoria` e `blocoRotina` apareciam ANTES de `blocoAgenda` — a Flora lia texto narrativo ("academia às 15h, trabalho das 6h às 15h") como contexto primário e depois lia a agenda estruturada, sem saber qual prevalecia.

3. **Tarefas invisíveis para a Flora.** `planoAtual.tarefas` nunca era injetado no `blocoAgenda`. A Flora não sabia quais tarefas com prazo existiam em cada dia, causando sugestões conflitantes.

4. **INICIO dos horários livres hardcoded em 6h.** `_horariosLivres` usava `INICIO = 6 * 60` independente do `HORA_INI = 1` configurado no calendário visual — inconsistência entre o que o usuário via no app e o que a Flora entendia como horário disponível.

**Impacto:** Flora mencionava compromissos inexistentes, colocava eventos em dias errados, e não sabia sobre tarefas com prazo. Confiança do usuário no app comprometida.

**Solução aplicada** (4 correções cirúrgicas em `flora.js`):

1. **REGRA CRÍTICA — SOURCE OF TRUTH** adicionada após as REGRAS ABSOLUTAS do `blocoAgenda`:
   - "A agenda estruturada acima é a ÚNICA fonte de verdade"
   - "NUNCA invente, assuma ou transfira compromissos que não estão explicitamente listados"
   - "O blocoRotina abaixo é CONTEXTO HISTÓRICO — NÃO é agenda atual"
   - "Antes de afirmar qualquer compromisso, verifique: esse evento está listado na agenda com esse dia e horário?"

2. **Reordenação dos blocos** no system prompt:
   - Antes: `blocoGamificacao → blocoMemoria → blocoRotina → blocoAgenda → blocoContextoTemporal`
   - Depois: `blocoGamificacao → blocoAgenda → blocoContextoTemporal → blocoMemoria → blocoRotina`
   - Agenda estruturada agora tem precedência visual e contextual sobre memória narrativa.

3. **Tarefas injetadas no blocoAgenda** por dia:
   - `planoAtual?.tarefas` filtradas por `t.prazo === dataYMD && !t.concluida`
   - Formato: `📝 Título da tarefa [TAREFA — blocoSugerido?]` abaixo dos compromissos do dia
   - Flora agora vê tarefas e compromissos do mesmo dia no mesmo contexto

4. **INICIO dos horários livres corrigido:** `INICIO = 6 * 60` → `INICIO = 1 * 60`
   - Agora bate com o calendário visual (`HORA_INI = 1` em `RoutineView.jsx`)

**Risco futuro:**
- O `blocoRotina` ainda é injetado no prompt (após a agenda) como contexto de hábitos. Se a Flora tiver dificuldade para distinguir "hábito histórico" vs "compromisso atual" mesmo com a regra SOURCE OF TRUTH, considerar removê-lo ou reformulá-lo para tornar explícito que é referência, não agenda.
- Tarefas sem prazo (`t.prazo = null`) continuam invisíveis na agenda — comportamento intencional (tarefas sem prazo não têm dia específico). Considerar injetar lista de tarefas sem prazo em bloco separado no futuro.

### [2026-05-24] BUG-010 — Compromissos adicionados/editados/deletados manualmente não persistiam no Supabase

**Sintoma:** Usuário adiciona, edita ou deleta compromisso diretamente pelo calendário. Ao reabrir o app ou logar em outro dispositivo, a alteração some.

**Causa raiz:** `adicionarCompromisso`, `editarItem` e `deletarItem` em `App.jsx` apenas faziam `setPlano + ls_set` — sem chamar nenhum endpoint. O endpoint `/api/usuario/salvar-plano` existia mas nunca era chamado por essas funções. Na próxima sessão, `carregarDadosUsuario` sobrescrevia o localStorage com o Supabase (que não tinha as mudanças manuais).

**Impacto:** Toda alteração manual no calendário (não feita via Flora) era perdida cross-device ou após reload com sessão nova.

**Solução aplicada:**
1. Nova função `salvarPlanoNoSupabase(novoPlano)` em `App.jsx` — fire-and-forget, não bloqueia UI.
2. `adicionarCompromisso`, `editarItem` e `deletarItem` agora chamam `salvarPlanoNoSupabase(novo)` após `ls_set`.
3. Endpoint `/api/usuario/salvar-plano` já existia em `server/index.js` — nenhuma alteração no backend necessária.

**Risco futuro:** Se o usuário fizer várias edições rápidas em sequência (ex: editar 5 compromissos em 2 segundos), serão disparadas 5 requisições paralelas ao Supabase. Por ser upsert, a última wins — sem risco de corrupção, mas pode gerar carga desnecessária. Considerar debounce futuramente.

### [2026-05-24] BUG-012 — Modal de edição de compromisso sem campo de horário final

### [2026-05-24] BUG-013 — Ritual de fechamento disparava duplicado e tag vazava para o usuário

**Sintoma A:** Tag `[RITUAL_FECHAMENTO]` aparecia como balão do usuário no chat (texto literal com o tag).
**Sintoma B:** Ao recarregar a página entre 20h e 23h sem completar o check-in, o ritual disparava novamente.

**Causa raiz A:** `enviarMensagem` adicionava `msgUser = { texto: input }` ao chat sem filtrar mensagens de sistema. A tag virava um balão visível para o usuário.

**Causa raiz B:** `checkinDisparadoRef = useRef(false)` era memória volátil da sessão React — resetava a cada reload. Sem persistência, a única proteção era `memoria.checkIns` (que só é populada após o check-in ser concluído). Se o usuário não completava o ritual, o dispatch ocorria novamente na próxima sessão.

**Solução aplicada:**
1. **Filtro de sistema:** `ehMensagemSistema = /^\[(RITUAL_FECHAMENTO|ROTINA_RETORNO|Sistema|MODO_CAOS)\]/.test(input)` — se verdadeiro, `novasMensagens` não inclui o balão do usuário. Cobre todos os 4 tipos de mensagem interna.
2. **Flag persistida:** Substituído `checkinDisparadoRef` por funções `jaDisparouCheckinHoje()` e `marcarCheckinDisparadoHoje()` que usam `localStorage.setItem('fluxo_checkin_disparado_YYYY-MM-DD', 'true')`. A flag sobrevive a reloads. `marcarCheckinDisparadoHoje` também limpa flags de dias anteriores automaticamente.

**Risco futuro:** Mensagens `[MODO_CAOS]` agora também são ocultas da UI — o usuário vê apenas a resposta da Flora, sem o trigger de sistema. Isso é comportamento correto, mas se alguma mensagem de sistema precisar de feedback visual no futuro, a regex precisará ser revisitada.

### [2026-05-26] BUG-014 — Flora reagendando compromissos pontuais automaticamente

**Sintoma:** Compromissos pontuais (com data específica, sem recorrência) marcados como concluídos reapareciam na semana seguinte com data avançada, sem o usuário pedir reagendamento.

**Causa raiz:** A função `ajustarDatasFuturas` em `server/index.js` avança datas passadas para a próxima ocorrência futura. Combinado com a ausência de uma regra explícita no system prompt proibindo reagendamento automático, a Flora podia interpretar compromissos pontuais passados como se fossem recorrentes e devolver o plano com datas avançadas na próxima resposta.

**Impacto:** Compromissos únicos (ex: consulta médica, reunião específica) ressurgiam na agenda em datas incorretas — causando confusão e perda de confiança do usuário no app.

**Solução aplicada:** Adicionado bloco `REGRA SOBRE REAGENDAMENTO — OBRIGATÓRIA` em `server/services/flora.js`, inserido antes da seção `REMOÇÃO PONTUAL vs REMOÇÃO DA ROTINA`. A regra proíbe explicitamente:
- Mover compromisso passado para data futura sem pedido explícito
- Interpretar compromisso pontual como recorrente
- Criar nova ocorrência de compromisso pontual
Reagendamento só ocorre quando o usuário usar palavras explícitas: "reagenda", "move", "adiar", "remarcar", "muda pra", "transfere pra".

**Risco futuro:** A função `ajustarDatasFuturas` no backend ainda opera sobre todas as datas do plano retornado pela Flora. Se a Flora retornar o plano sem modificar datas (apenas com plano: null), não há problema. Mas se ela retornar o plano completo após uma conversa que não envolveu alterações, `ajustarDatasFuturas` pode ainda avançar datas passadas. Considerar proteger compromissos concluídos (com `concluida: true`) de `ajustarDatasFuturas` no futuro.
